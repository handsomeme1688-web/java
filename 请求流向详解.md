# 请求流向详解——以 day4 的两条真实请求为例

> 配合聊天里那张图读，图上的编号和这里的步骤号一一对应。

## 第 0 步：启动时发生了什么（最容易被忽略的一段）

`Day4Application.main()` 运行时，Spring 扫描 `com.zoee.day4` 下所有类，**把带注解的类各造一个对象放进容器**：

| 类 | 注解 | 谁用它 |
|---|---|---|
| UserController | @RestController | DispatcherServlet 按路由表调它 |
| UserServiceImpl | @Service | 注入给 Controller 的构造器 |
| UserMapper（代理） | @Mapper | MP 生成实现类，注入给 Service |
| LoginCheckInterceptor | @Component | WebConfig 把它挂到拦截链上 |
| WebConfig | @Configuration | 启动时执行一次 addInterceptors |
| GlobalExceptionHandler | @RestControllerAdvice | DispatcherServlet 出异常时查它 |

记住：**所有对象启动时就造好了，请求来了只是"用"，没有任何 new。**
然后内置 Tomcat 开始监听 8080 端口。

---

## 请求一：POST /users/login（白名单请求）

**1. Postman → Tomcat**
Postman 把 HTTP 报文发到 8080：请求行 `POST /users/login`、头 `Content-Type: application/json`、体 `{"username":"tom","password":"123456"}`。
Tomcat 收到后封装成两个对象：`HttpServletRequest`（请求的一切）和 `HttpServletResponse`（往里写响应），转交 **DispatcherServlet**（Spring MVC 的总调度，所有请求的唯一入口）。

**2. DispatcherServlet 查路由 + 过安检**
拿 "POST + /users/login" 查路由表（启动时由 @RequestMapping/@PostMapping 注册的），命中 `UserController.login` 方法。
执行拦截器链：`/users/login` 在 WebConfig 的 excludePathPatterns 里 → **跳过拦截器**。

**3. 参数装配（@RequestBody 的工作时刻）**
框架看到 login 的参数 `@RequestBody User user`，就叫 Jackson 把 JSON 字符串反序列化成 User 对象——只有 username/password 有值，其余字段是 null。
> 谁调用：框架。传给谁：`login(User user)` 的形参。

**4. Controller → Service**
`login` 方法体执行 `userService.login(user)`。
> 调用者：UserController。参数：User（裸数据）。userService 是启动时注入的 UserServiceImpl 实例。

**5. Service 干业务（今天的主战场）**
`UserServiceImpl.login(User user)` 内部三段：
- a. `userMapper.selectOne(wrapper)` → 参数是查询条件（username='tom'）；**Mapper 代理**把它翻译成 SQL `SELECT ... FROM user WHERE username=?`，经 JDBC 发给 MySQL；结果行被映射回 **User 对象**（existUser，全字段），查不到返回 **null**
- b. 校验：existUser 为 null → `throw new UserException("用户名不存在")`——方法立刻中断，异常向上飞（见第 7 步）；BCrypt.checkpw 比对失败同理
- c. 校验通过：组 `Map claims = {id: existUser.getId()}` → `JwtUtils.generateJwt(claims)` → 返回 **String token**
- `return token` → 返回给 Controller

**6. 回程：对象一路变成 JSON**
Controller 拿到 token，包信封：`Result.success(token)` → 返回 **Result 对象**。
DispatcherServlet 拿到 Result，叫 Jackson **序列化**成 JSON 字符串，写进 HttpServletResponse → Tomcat 发回 Postman。
你看到的 `{"code":1,"data":"eyJ...","msg":"success"}` 就是 Result 对象的 JSON 化。

**7. 异常路（如果第 5b 抛了）**
UserException 从 Service 飞出，穿过 Controller（它没 catch，异常直接穿透），落到 DispatcherServlet 手里 → 它去查 @RestControllerAdvice → 命中 `GlobalExceptionHandler.ex(UserException e)` → 该方法返回 `Result.error(e.getMessage())` → 一样序列化返回。
**这就是 Controller/Service 里永远不写 try-catch 的原因：楼下有人接。**

## 请求二：GET /users/me（受保护请求）

**1.** Postman 发 `GET /users/me`，Header 里带 `token: eyJ...`。
**2.** DispatcherServlet 查路由命中 `me()`；`/users/me` 不在白名单 → **进拦截器**。
**3.** 框架调用 `preHandle(request, response, handler)`（三个参数都是框架给的）：
- `request.getHeader("token")` → 取出字符串；没有 → `response.setStatus(401); return false` → **流程终止，Controller 不会执行**
- `JwtUtils.parseJwt(token)` → 用密钥验签+解析 → 返回 **Claims**（一个 Map）；令牌被改过/过期/是旧密钥签的 → 抛异常 → catch → 401，同上终止
**4.** 验签成功：`claims.get("id")` → `UserContext.setUserId(id)` → 存进**当前线程**的 ThreadLocal 储物格。
**5.** `return true` → 放行，DispatcherServlet 这才调用 `me()`。
**6.** `me()` 没有参数——它要的身份不来自请求，来自储物格：`UserContext.getUserId()`。**同一次请求全程同一条线程**，所以取到的正是第 4 步存的那个 id。
**7.** `Result.success(id)` → 序列化 → 返回。
**8.** 响应发出后，框架回头调 `afterCompletion` → `UserContext.remove()` 清格子（线程要还给线程池给下一个用户用，不清会串号）。

## 接到新需求时，怎么决定"建什么类、加什么注解、写什么方法"

按这张决策表自问（顺序固定）：

| 自问 | 答"是"就做 | 注解 | 参数/返回值规矩 |
|---|---|---|---|
| 要有新的 URL 吗？ | Controller 加方法 | @GetMapping/@PostMapping + 参数上按传参方式选 @RequestBody/@PathVariable/@RequestParam | 参数=前端给的数据；返回值一律 `Result<X>` |
| 有业务规则/判断吗？ | Service 接口+impl 各加方法 | 无（类上已有 @Service） | 参数和返回值都是**裸数据**（User/Long/List），永不碰 Result 和 request |
| 白送的 CRUD 不够用吗？ | Mapper 加方法 | @Select 等（类上已有 @Mapper） | 参数=SQL 占位符要的值；返回=实体/List |
| 这规则是不是**所有接口**都要过？ | 拦截器/异常处理器/配置类 | @Component / @RestControllerAdvice / @Configuration | 一次性基建，写完基本不再动 |
| 只是个纯计算，谁都可能用？ | utils 加静态方法 | 无注解 | static，不依赖任何 Bean |

三条铁律收尾：
1. **依赖方向永远单向**：Controller→Service→Mapper，不许反向、不许跳层
2. **每层只说自己的语言**：Controller 说 HTTP（Result/状态码），Service 说业务（裸对象/异常），Mapper 说 SQL
3. **写不出某个方法时，先写它的调用方**——你自然就知道它该收什么、还什么
