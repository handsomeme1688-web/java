# 后端开发心法（写给第 4 天的自己，Day 21 回来重读一遍）

## 一、一张地图：所有功能都住在同一条线上

```
浏览器/Postman → 拦截器 → Controller → Service → Mapper → 数据库
                                ↘ 任何一层抛异常 → GlobalExceptionHandler → Result.error
```

后端 90% 的日常开发，就是往这条线上"加站点"或"改站点"。接到任何需求，第一个动作不是写代码，
是问自己：**这个需求要动这条线上的哪几站？**
- 加个查询接口 → Controller + Service + Mapper 各加一段
- 加登录校验 → 拦截器这一站
- 改返回格式 → Result / VO 这一层
迷路的时候，回这张图找自己的位置。

## 二、一个循环：小步快跑（今天所有痛苦的解药）

**写最小的一步 → 跑 → 看结果 → 再写下一步。永远不要写完五个文件才第一次运行。**

今天你一口气写完拦截器+配置+工具类+上下文才启动，于是五个错叠在一起爆炸，
拆弹拆了一晚上。如果每写完一个文件就跑一次，每个错都会在它出生的那一刻被抓住。

## 三、新接口固定七步（照抄，直到形成肌肉记忆）

1. **纸上先写接口长相**：方法+URL+入参 JSON+出参 JSON（写不出来=需求没想清，别碰键盘）
2. 判断动不动表/实体
3. Controller 写空方法返回假数据 → **跑**，Postman 通了路由
4. Service 写真逻辑 → **跑**
5. 需要自定义 SQL 才动 Mapper → **跑**，看控制台 SQL
6. 补异常分支：什么情况该抛 UserException → **跑**，看报错是不是 Result 形状
7. Postman 一正一反 + 测试类两条（存在/不存在），全绿才算完

## 四、怎么知道"干了什么、没干什么、下一步干什么"

代码不靠脑子记，靠清单：
- **开工 3 行**：今天要跑通什么（例：注册/登录/me 三接口 Postman 全通）
- 把任务拆成 5~8 步写下来（按"被依赖的先写"排序），做一步勾一步
- **收工 3 行**：跑通了什么 / 卡在哪 / 明天第一件事
手册每天 21:40 的日志就是干这个的，别把它当作业，它就是"下一步做什么"的答案本身。

## 五、怎么"不出错"——错误的正确打开方式

不出错不可能，高手只是**恢复得快**。恢复快靠的全是今天你练过的招：
- Postman 先看**状态码**：200 成功｜401 拦截器拦了｜404 路径不对｜405 方法不对｜500 代码炸了
- 服务端问题看 **IDEA 控制台**，红字翻到最底下的 `Caused by` 就是真凶
- 看不出来就**装探针**：println 打印"代码走到哪了、变量是什么"，让代码自己交代
- 字符串配置（路径/包名/yml 键）出问题**不报错、只失效**，第一反应查字符级差异
- catch 里永远别静默吞异常，至少打印出来

## 六、怎么变流畅——只有一个字：重复

第一遍抄着写（今天）→ 第二遍不看资料重写（手册 Day5"删掉重写"就是为这设计的）→
第三遍才长在你手上。明天把 day4 删了凭记忆重写一遍，卡住才翻旧代码。
你会震惊于第二遍只要一半时间——那个提速的感觉，就是"流畅"在生长。

## 七、Day 4 的你已经会的东西（低落的时候看这段）

三层架构、统一返回、全局异常、BCrypt、JWT 签发与校验、拦截器、ThreadLocal、
环境变量管密钥、Postman 调试、看日志抓 Caused by、探针排查法。
四天前你连构造方法都没认出来。曲线是陡的，但你在往上走。
