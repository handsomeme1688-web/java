# 八股高频背诵表（150 题）

> 来源：51 天手册全部"背"条目 + 每题背后必然出现的追问层。
> 用法：遮住答，对空气讲，讲得出且能扛一层追问才算过。没学到的板块先跳过，学到当天回来背。

## 一、Java 基础

1.问：== 和 equals() 的区别？
答：== 对基本类型比值，对引用类型比内存地址；equals() 是 Object 的方法，默认也是比地址，但 String 等类重写后比内容。比较字符串内容必须用 equals()。

2.问：为什么重写 equals() 必须重写 hashCode()？
答：规约要求"equals 相等的两个对象 hashCode 必须相等"。只重写 equals 不重写 hashCode，两个"相等"的对象哈希值不同，放进 HashMap/HashSet 会被分到不同的桶，出现"同一个对象存两份"的诡异 bug。

3.问：String 为什么设计成不可变？
答：三点：字符数组 private final 且不提供修改方法，类被 final 修饰防子类破坏；好处是线程安全、可放心做常量池共享、hashCode 可缓存（适合当 Map 的 key）。

4.问：字符串常量池是干什么的？
答：JVM 为避免字符串重复创建开的一块区域。字面量赋值（String s = "ab"）会先查池，有就复用引用，没有才创建。所以 "ab"=="ab" 为 true。

5.问：String s = new String("abc") 创建了几个对象？
答：1 个或 2 个。常量池没有 "abc" 时创建 2 个（池里一个、堆里一个）；已有时只在堆里创建 1 个。

6.问：String、StringBuilder、StringBuffer 区别？
答：String 不可变，每次"修改"都生成新对象；StringBuilder 可变、非线程安全、性能最好；StringBuffer 可变、方法加了 synchronized 线程安全但慢。循环拼接字符串必须用 StringBuilder。

7.问：重载和重写的区别？
答：重载（Overload）是同一个类里方法同名、参数列表不同，编译期决定调哪个；重写（Override）是子类覆盖父类方法，方法签名相同，运行期根据实际对象决定（多态的基础）。

8.问：构造方法有什么特点？能被重写吗？
答：与类同名、没有返回值类型（连 void 都没有）、new 时自动执行；不能被重写（构造器不继承），但可以重载。写了有参构造后默认无参构造消失，需要手动补。

9.问：泛型是什么？解决什么问题？
答：把类型参数化，编译期就拦住类型错误——往 List<User> 里塞 String 直接编译不过，取出来也不用强转。运行期泛型会被擦除。

10.问：受检异常和非受检异常的区别？
答：受检（Checked）继承 Exception，编译器强制处理（try-catch 或 throws），如 IOException；非受检（Unchecked）继承 RuntimeException，编译器不管，如 NPE、我们自定义的业务异常。

11.问：finally 一定会执行吗？它常用来干什么？
答：除了 System.exit() 或 JVM 崩溃，finally 一定执行——即使 try 里有 return。用途是释放资源（关连接、关流、解锁）。

12.问：接口和抽象类怎么选？
答：接口定义"能做什么"（行为约束，可多实现）；抽象类定义"是什么"（代码复用，单继承）。要共享字段和方法实现选抽象类，只约束行为选接口。

13.问：深拷贝和浅拷贝的区别？
答：浅拷贝复制对象本身，内部引用类型字段仍指向同一个对象（改一个影响另一个）；深拷贝连内部对象也复制一份，完全独立。

14.问：Java 是值传递还是引用传递？
答：只有值传递。传对象时传的是"引用的副本"——方法内能通过它修改对象内容，但让形参指向新对象不影响实参。

15.问：Stream 的 filter/map/collect/groupingBy 各干什么？
答：filter 按条件筛选留下想要的；map 把每个元素变形（如 User→username）；collect 把流收集回容器（toList/toMap）；groupingBy 按字段分组，得到 Map<分组键, List>。

## 二、集合

16.问：ArrayList 和 LinkedList 的区别？
答：ArrayList 底层数组，随机访问 O(1)，中间增删要挪元素 O(n)；LinkedList 底层双向链表，随机访问要遍历 O(n)，只有已持有节点引用时增删才是 O(1)。日常 99% 场景用 ArrayList。

17.问：ArrayList 的扩容机制？
答：默认容量 10，放不下时扩容为原来的 1.5 倍（oldCapacity + oldCapacity>>1），把旧数组拷贝到新数组。能预估大小就用 new ArrayList<>(n) 避免反复扩容。

18.问：HashMap 的底层结构？
答：JDK8 是数组+链表+红黑树。put 时对 key 的 hashCode 扰动后取模定位桶；冲突了挂链表；链表长度≥8 且数组长度≥64 才转红黑树，查询从 O(n) 降到 O(logn)。

19.问：为什么链表≥8 且数组≥64 才转红黑树？
答：泊松分布下单桶到 8 个节点的概率极低（千万分之一级），到 8 说明哈希冲突异常严重才值得转；数组太小时冲突多是容量问题，优先扩容而不是转树。

20.问：HashMap 为什么扩容 2 倍？长度为什么是 2 的幂？
答：长度是 2 的幂时，取模可以用位运算 (n-1)&hash 代替 %，快；且扩容后元素要么留在原位要么移动"原位+旧容量"，不用重新计算哈希。

21.问：负载因子为什么默认 0.75？
答：空间和时间的折中：太大（如 1）冲突概率高、链表长、查询慢；太小（如 0.5）频繁扩容浪费空间。0.75 是实践得出的平衡点。

22.问：HashMap 线程安全吗？多线程下用什么？
答：不安全，并发 put 会丢数据（JDK7 还会扩容成环死循环）。多线程用 ConcurrentHashMap，不用 Hashtable（全表一把锁，太慢）。

23.问：ConcurrentHashMap 怎么保证线程安全？
答：JDK8 用 CAS + synchronized 锁单个桶的头节点——锁粒度细到桶级别，不同桶并发写互不阻塞；读操作基本无锁（volatile 保证可见性）。

24.问：HashSet 怎么去重？
答：底层就是 HashMap（元素当 key）。先比 hashCode 定桶，再在桶内用 equals 精确比较——所以自定义对象去重必须同时重写这两个方法。

## 三、并发

25.问：进程和线程的区别？
答：进程是资源分配的最小单位（独立内存空间）；线程是 CPU 调度的最小单位，同进程的线程共享内存。线程切换开销远小于进程。

26.问：创建线程有哪几种方式？
答：继承 Thread、实现 Runnable、实现 Callable（有返回值可抛异常）、线程池提交任务。实际开发只用线程池。

27.问：volatile 的作用？
答：两个：保证可见性（写立刻刷主存，读强制从主存取）；禁止指令重排序（内存屏障）。但不保证原子性——volatile 修饰的 i++ 照样出错，原子性要靠 synchronized 或 AtomicInteger。

28.问：DCL 双重检查锁单例为什么要加 volatile？
答：new 对象分三步：分配内存→初始化→引用赋值。不加 volatile 可能重排成 1→3→2，另一个线程在第 3 步后拿到"未初始化完的对象"。volatile 禁止这个重排。

29.问：synchronized 的三种用法分别锁什么？
答：修饰实例方法锁当前对象 this；修饰静态方法锁类的 Class 对象；修饰代码块锁括号里指定的对象。静态和非静态 synchronized 方法互不互斥（锁不同）。

30.问：synchronized 的底层原理？
答：代码块靠 monitorenter/monitorexit 指令，方法靠 ACC_SYNCHRONIZED 标志，本质都是获取对象监视器 monitor——每个对象头里都关联一个，拿到才能进，出来释放。

31.问：synchronized 锁升级过程？
答：无锁→偏向锁→轻量级锁（CAS 自旋）→重量级锁（挂起线程）。竞争逐渐激烈逐级升、不可降。注意：偏向锁 JDK15 默认关闭、JDK18 已废弃，只作历史背景讲。

32.问：synchronized 和 ReentrantLock 的区别？
答：四点：①实现层面——JVM 关键字 vs JDK API（需手动 unlock，放 finally）；②ReentrantLock 支持等待可中断 lockInterruptibly；③支持公平锁；④支持多个 Condition 精准唤醒和 tryLock 超时。功能不够用才选 ReentrantLock。

33.问：公平锁和非公平锁？
答：公平锁先来先得，上下文切换多、吞吐低；非公平锁允许插队，性能好但可能线程饥饿。ReentrantLock 默认非公平。

34.问：CAS 的原理？
答：Compare And Swap：拿期望值 E 和内存当前值 V 比较，相等才把 V 更新为新值 N，不等就失败重试（自旋）。是一条 CPU 原子指令，无锁但线程安全，Atomic 类的底层。

35.问：CAS 的 ABA 问题怎么解决？
答：值从 A 改成 B 又改回 A，CAS 察觉不到。解法是加版本号——每改一次版本+1，比较"值+版本"，JDK 的 AtomicStampedReference 就是这个思路，数据库乐观锁的 version 字段同理。

36.问：AQS 是什么？
答：AbstractQueuedSynchronizer，JUC 锁的地基：一个 volatile int state 表示资源状态 + 一个 CLH 双向队列管理抢不到资源的线程。ReentrantLock、CountDownLatch 都基于它实现。

37.问：乐观锁和悲观锁怎么选？
答：悲观锁（synchronized、SELECT FOR UPDATE）假设必冲突，先加锁，适合写多；乐观锁（CAS、版本号）假设少冲突，提交时校验，适合读多写少。写竞争激烈时乐观锁会疯狂重试反而更慢。

38.问：ThreadLocal 的原理？
答：每个 Thread 内部有一张 ThreadLocalMap，key 是 ThreadLocal 对象，value 是存的值——数据存在线程自己身上，天然隔离。典型用途：登录拦截器存当前 userId，同请求内随处可取。

39.问：ThreadLocal 为什么会内存泄漏？怎么办？
答：ThreadLocalMap 的 key 是弱引用会被回收，value 是强引用还挂在线程上；线程池里线程不死，value 就一直泄漏，还会串到下一个请求。所以用完必须 remove()——放拦截器的 afterCompletion 里。

40.问：线程池的 7 个参数？
答：corePoolSize 核心线程数、maximumPoolSize 最大线程数、keepAliveTime+unit 非核心线程空闲存活时间、workQueue 任务队列、threadFactory 线程工厂、handler 拒绝策略。

41.问：线程池的执行流程？
答：来任务先用核心线程；核心满了进队列；队列满了开非核心线程到最大值；还满就走拒绝策略。口诀：核心→队列→非核心→拒绝。

42.问：四种拒绝策略？
答：AbortPolicy 直接抛异常（默认）；CallerRunsPolicy 谁提交谁自己跑（降速保命）；DiscardPolicy 静默丢弃；DiscardOldestPolicy 丢队列最老的再重试。

43.问：线程数怎么定？
答：不死背 2N。理论公式 Ncpu × 目标利用率 × (1 + 等待时间/计算时间)：CPU 密集≈N+1，IO 密集等待占比高所以≈2N 起步，最终必须压测定，说得出"为什么"。

44.问：execute 和 submit 的区别？
答：execute 只接 Runnable 无返回值，异常直接抛到线程把线程干掉；submit 接 Callable/Runnable 返回 Future，异常被吞进 Future，get() 时才抛出——不 get 就静默丢失，要小心。

45.问：为什么不建议用 Executors 创建线程池？
答：newFixedThreadPool 用无界队列，任务堆积会 OOM；newCachedThreadPool 最大线程数是 Integer.MAX_VALUE，线程爆炸也会 OOM。阿里规约要求手动 new ThreadPoolExecutor，7 个参数自己定。

## 四、JVM

46.问：JVM 运行时内存区域有哪些？
答：线程共享：堆（对象）、方法区/元空间（类信息、常量）；线程私有：虚拟机栈（方法栈帧、局部变量）、本地方法栈、程序计数器（记录执行到哪行字节码）。

47.问：对象创建的 5 步？
答：类加载检查→分配内存（指针碰撞/空闲列表）→零值初始化→设置对象头（类指针、哈希、锁标志）→执行构造方法 init。

48.问：怎么判断对象可以被回收？
答：可达性分析：从 GC Roots（栈里的局部变量、静态变量、常量、JNI 引用）出发往下搜，搜不到的对象不可达，判死。不用引用计数因为解决不了循环引用。

49.问：三种垃圾回收算法？
答：标记-清除（快但产生碎片）；标记-复制（无碎片但浪费一半空间，适合存活率低的新生代）；标记-整理（无碎片不浪费但移动成本高，适合老年代）。

50.问：为什么分代收集？
答：绝大多数对象朝生夕死。新生代（Eden+两个 Survivor）用复制算法频繁小规模回收（Minor GC）；熬过多次的进老年代，用标记-整理低频回收。分代让每代用最适合的算法。

51.问：G1 收集器一句话？
答：把堆切成等大的 Region，维护每个 Region 的回收收益，优先回收"垃圾最多"的（Garbage First），可以设定停顿时间目标，兼顾吞吐和低延迟，JDK9+ 默认。

52.问：类加载的过程？
答：加载（读字节码生成 Class 对象）→验证（校验格式安全）→准备（静态变量分配内存赋零值）→解析（符号引用转直接引用）→初始化（执行静态代码块和静态赋值）。

53.问：什么是双亲委派？有什么好处？
答：类加载器收到请求先委托父加载器，父加载不了才自己来（Bootstrap→Ext/Platform→App）。好处：①核心类不被篡改（自己写的 java.lang.String 不会被加载）②避免类重复加载。

54.问：怎么破坏双亲委派？
答：重写 ClassLoader 的 loadClass（而不是 findClass）；典型场景：JDBC 的 SPI 用线程上下文类加载器反向委派、Tomcat 每个 webapp 独立类加载器实现隔离、热部署。

55.问：反射是什么？@Autowired 为什么能生效？
答：反射=运行期动态获取类信息并操作对象（Class/Method/Field）。Spring 启动时扫描注解，用反射创建对象、遍历字段，发现 @Autowired 就从容器找到对应 Bean 后 field.set() 塞进去——注解+反射=框架魔法的本体。

## 五、MySQL

56.问：索引为什么用 B+树不用 B 树？
答：B+树非叶子节点只存索引不存数据，一页能放更多键，树更矮 IO 更少；数据全在叶子且叶子间有双向链表，范围查询和排序直接顺着链表扫，B 树做不到。

57.问：为什么不用哈希索引和红黑树？
答：哈希等值查询 O(1) 但不支持范围查询和排序；红黑树是二叉树，千万数据高度几十层，磁盘 IO 次数太多。B+树矮胖（3~4 层扛千万行），专为磁盘设计。

58.问：聚簇索引和二级索引的区别？
答：聚簇索引叶子节点存整行数据（主键索引就是，表本身按它组织）；二级索引叶子只存"索引列+主键值"。一张表只有一个聚簇索引。

59.问：什么是回表？
答：走二级索引查到主键后，还要拿主键回聚簇索引再查一次拿整行。二级索引查两棵树，这就是"回表"，能避免就避免。

60.问：什么是覆盖索引？
答：查询的列全部包含在索引里（SELECT id,name 而索引就是 (name)），二级索引一棵树搞定不用回表。这是"不要 SELECT *"的核心理由之一。

61.问：最左前缀原则？
答：联合索引 (a,b,c) 相当于按 a→b→c 排序，查询条件必须从最左列连续命中：a、ab、abc 能走索引；b、bc、ac（c 断了）不能完整使用。

62.问：哪些写法会让索引失效？
答：对索引列做函数或计算、隐式类型转换（字符串列传数字）、左模糊 LIKE '%x'、OR 连接了非索引列、不符合最左前缀。不背结论，用 EXPLAIN 验证过才作数。

63.问：EXPLAIN 主要看哪几列？
答：type 访问级别（const>ref>range>index>ALL，ALL 是全表扫描要警惕）；key 实际用的索引；rows 预估扫描行数；Extra 里 Using index=覆盖索引好，Using filesort=文件排序坏。

64.问：什么情况不建议建索引？
答：区分度低的列（性别）、频繁更新的列（维护成本）、表很小、写多读少的表。索引不是白来的：占空间且拖慢增删改。

65.问：ACID 分别靠什么保证？
答：原子性靠 undo log（回滚日志）；持久性靠 redo log（先写日志再刷盘）；隔离性靠 MVCC + 锁；一致性是前三者共同达成的结果。

66.问：四种隔离级别分别解决什么问题？
答：读未提交（啥也不解决，有脏读）→读已提交 RC（解决脏读）→可重复读 RR（再解决不可重复读，MySQL 默认）→串行化（再解决幻读，性能最差）。

67.问：脏读、不可重复读、幻读分别是什么？
答：脏读=读到别人未提交的数据；不可重复读=同一事务内两次读同一行值变了（别人改了并提交）；幻读=两次同条件查询行数变了（别人插入了新行）。

68.问：MVCC 的原理？
答：每行带隐藏字段（事务id、回滚指针）形成 undo log 版本链；事务读取时生成 ReadView（含活跃事务列表），按可见性规则沿版本链找自己"该看见"的版本——读不加锁，读写不阻塞。

69.问：RC 和 RR 生成 ReadView 的区别？
答：RC 每次查询都生成新 ReadView（所以能看到别人新提交的，不可重复读）；RR 只在第一次查询生成、整个事务复用（所以可重复读）。一句话记：RC 每次、RR 一次。

70.问：RR 级别完全解决幻读了吗？
答：没有。快照读靠 MVCC 防住了；当前读（SELECT FOR UPDATE、UPDATE）靠临键锁/间隙锁防插入。但"先快照读再当前读"的混用场景仍可能出现幻读，所以说 RR 只是很大程度上避免。

71.问：MySQL 的锁分几层？行锁有哪几种？
答：全局锁（备份用）、表级锁（表锁/元数据锁/意向锁）、行级锁。行锁三种：记录锁锁一行、间隙锁锁开区间防插入、临键锁=记录+间隙（左开右闭，RR 默认）。

72.问：一条 SQL 很慢，怎么排查？（完整口述）
答：先开慢查询日志定位到具体 SQL → EXPLAIN 看执行计划（type 是不是 ALL、key 用没用上、rows 多大）→ 分析原因：缺索引就加、索引失效就改写 SQL、数据量问题就分页/限定范围 → 改完再 EXPLAIN + 压测对比验证。

73.问：一条 UPDATE 语句的执行流程？
答：执行器调存储引擎定位到行→先写 undo log（备回滚）→内存中改数据页（Buffer Pool）→写 redo log 处于 prepare 状态→写 binlog→提交事务，redo log 改 commit 状态（两阶段提交）→后台异步刷脏页到磁盘。

74.问：为什么需要两阶段提交？
答：redo log（InnoDB 的）和 binlog（Server 层的）是两份日志，必须保持一致。若先写完一个就崩溃，主库恢复和从库复制的数据会对不上。两阶段提交用 prepare+commit 保证两份日志"要么都有效要么都作废"。

75.问：redo log 和 binlog 的区别？
答：redo log 是 InnoDB 独有的物理日志（页改了什么），循环写，用于崩溃恢复；binlog 是 Server 层的逻辑日志（语句/行变更），追加写，用于主从复制和数据恢复。

76.问：主键为什么推荐自增，不用 UUID？
答：聚簇索引按主键有序存放。自增主键永远往末尾追加，页写满就开新页；UUID 无序，插入位置随机导致频繁页分裂、碎片多、缓存命中差。

## 六、Redis

77.问：Redis 五种基本类型和使用场景？
答：String（缓存对象、计数器、分布式锁）；Hash（购物车、对象属性）；List（消息队列、时间线）；Set（点赞、共同关注、抽奖去重）；ZSet（排行榜、延迟队列）。

78.问：五种类型的底层结构？
答：String→SDS 动态字符串；List→quicklist；Hash→listpack（小）/哈希表（大）；Set→整数集合（全整数且少）/哈希表；ZSet→listpack（小）/跳表（大）。Redis 7.0 用 listpack 全面替代了压缩列表。

79.问：Redis 为什么快？
答：三点：纯内存操作；命令执行单线程无锁竞争、无上下文切换；IO 多路复用（epoll）单线程管理大量连接。注意："命令执行是单线程"≠"Redis 只有一个线程"——还有 AOF 刷盘、异步删除等后台线程，6.0 后网络 IO 也多线程了。

80.问：RDB 和 AOF 的区别？
答：RDB 是内存快照（二进制、恢复快、可能丢最后几分钟数据）；AOF 记录每条写命令（丢数据少、文件大恢复慢）。生产常用混合持久化：AOF 重写时前半段存 RDB、增量存 AOF，兼得两者。

81.问：什么是缓存穿透？怎么解决？
答：查询"数据库里根本不存在"的数据，缓存永远没有，请求全打到库（常是恶意攻击）。解法：缓存空值（短 TTL）、布隆过滤器在入口拦截"一定不存在"的 key、接口层参数校验。

82.问：什么是缓存击穿？怎么解决？
答：某个热点 key 过期瞬间，海量请求同时打到数据库。解法：互斥锁（只放一个线程去重建缓存，其他等待）或逻辑过期（永不过期+后台异步更新）。

83.问：什么是缓存雪崩？怎么解决？
答：大量 key 同一时间过期（或 Redis 挂了），请求洪峰全砸向数据库。解法：过期时间加随机值打散、热点数据不过期、多级缓存、Redis 高可用+限流降级兜底。

84.问：缓存和数据库双写，怎么保证一致性？
答：标准答案是"先更新数据库，再删除缓存"（Cache Aside）。先删缓存会在并发下把旧值读回来。补强手段：延迟双删对付"删后又被旧读回填"；要求更高就订阅 binlog（canal）异步删，保证最终一致。

85.问：为什么是删缓存而不是更新缓存？
答：更新缓存在并发下顺序错乱会留下旧值，且很多缓存是算出来的更新成本高、可能更新了也没人读（浪费）。删除是幂等的懒加载：下次读时再重建。

86.问：Redis 分布式锁怎么实现？
答：SET key 唯一值 NX EX 秒数——NX 保证只有一个客户端能设置成功，EX 防死锁，value 用 UUID 标识持有者；释放时用 Lua 脚本"先判断 value 是自己的再删"，保证判断+删除原子性。

87.问：分布式锁的三个坑？
答：①误删他人锁——value 存唯一标识+Lua 原子校验删除；②锁超时但业务没跑完——加看门狗自动续期（Redisson 已内置）；③不可重入——Redisson 用 Hash 结构记录持有者和重入次数。

88.问：Redis 的过期删除策略？
答：惰性删除（访问时发现过期才删，省 CPU 费内存）+ 定期删除（周期性随机抽查，过期比例超 25% 就继续抽）两者配合，平衡 CPU 和内存。

89.问：内存淘汰策略有哪些？
答：内存到 maxmemory 触发。八种：noeviction 拒绝写（默认）；volatile-lru/lfu/ttl/random 只淘汰有过期时间的；allkeys-lru/lfu/random 全体淘汰。缓存场景常用 allkeys-lru。

90.问：主从、哨兵、Cluster 各解决什么问题？
答：主从复制解决读扩展和数据备份（写主读从，异步复制）；哨兵解决主挂了没人管——监控+自动故障转移选新主；Cluster 解决单机容量上限——16384 个哈希槽把数据分片到多台。

## 七、Spring / 框架

91.问：IOC 是什么？
答：控制反转——对象的创建和依赖装配从"我 new"反转为"容器造好了给我"。好处：解耦（换实现不改调用方）、对象统一管理、方便测试时替换假对象。DI（依赖注入）是它的实现手段。

92.问：依赖注入有几种方式？为什么推荐构造器注入？
答：构造器注入、Setter 注入、字段注入(@Autowired)。推荐构造器：依赖可以 final（不可变不为空）、不依赖容器也能 new 出来测试、循环依赖会在启动时暴露而不是藏着。配 @RequiredArgsConstructor 一行不用多写。

93.问：Spring Bean 的生命周期？
答：实例化（构造器）→属性填充（注入依赖）→Aware 回调（感知容器）→BeanPostProcessor 前置处理→初始化（@PostConstruct/InitializingBean）→BPP 后置处理（AOP 代理在这里生成）→使用→容器关闭时销毁回调。

94.问：单例 Bean 有线程安全问题吗？
答：Bean 默认单例，所有请求共用一个实例。无状态（没有可变成员变量，如标准的 Controller/Service）就安全；有可变成员变量就不安全——要么改原型作用域，要么用 ThreadLocal，要么加锁。

95.问：AOP 是什么？用在哪？
答：面向切面编程——把日志、事务、权限这类横切逻辑从业务代码里抽出来，通过动态代理在方法前后织入。@Transactional、全局日志、接口耗时统计都是 AOP。

96.问：JDK 动态代理和 CGLIB 的区别？
答：JDK 代理基于接口（实现相同接口的代理类），目标必须有接口；CGLIB 生成目标类的子类覆盖方法，无接口也行，但 final 类/方法代理不了。Spring 默认：有接口用 JDK，没有用 CGLIB（Boot 里默认全走 CGLIB）。

97.问：三级缓存怎么解决循环依赖？
答：一级缓存放成品 Bean，二级放提前曝光的半成品，三级放对象工厂（能生成代理）。A 创建中就把工厂放三级，B 需要 A 时能拿到"提前引用"完成注入，A 再完成自己。注意：只能解决 setter/字段注入的循环，构造器循环依赖无解——设计上就该消除循环。

98.问：@Transactional 什么情况会失效？
答：高频八场景：方法不是 public；同类内部 this 调用（没走代理）；异常被 try-catch 吞了；抛的是受检异常没配 rollbackFor；类没被 Spring 管理；多线程里跑；传播行为配错；数据库引擎不支持事务（MyISAM）。核心原理都指向一件事：事务靠代理生效，绕过代理/异常没抛出去就失效。

99.问：事务传播行为 REQUIRED 和 REQUIRES_NEW 的区别？
答：REQUIRED（默认）：有事务就加入，没有就新建——外层回滚里层跟着回滚；REQUIRES_NEW：总是挂起当前事务另开新事务——里层独立提交，外层回滚不影响它（记日志常用）。

100.问：Filter 和 Interceptor 的区别？
答：Filter 是 Servlet 规范的，在 DispatcherServlet 之前执行，管所有请求，拿不到 Spring 上下文细节；Interceptor 是 Spring MVC 的，在 DispatcherServlet 之后、Controller 前后执行，能拿到 handler 信息、能注入 Bean。登录校验用 Interceptor 更顺手。

101.问：DispatcherServlet 的处理流程？
答：接收请求→HandlerMapping 查路由找到 Controller 方法→执行拦截器 preHandle→参数解析绑定→调用 Controller→返回值序列化（@ResponseBody→Jackson）→postHandle→渲染/写响应→afterCompletion。所有请求的总调度。

102.问：SpringBoot 自动配置的原理？
答：@SpringBootApplication 里的 @EnableAutoConfiguration 会读取依赖 jar 里 META-INF 的自动配置清单，按 @ConditionalOnClass/OnMissingBean 等条件判断"你引了什么依赖就给你配什么 Bean"——引 starter 就能用的原因。

103.问：MyBatis 里 #{} 和 ${} 的区别？
答：#{} 预编译占位符，参数作为数据传入（SQL 里是 ?），防注入；${} 字符串拼接，参数直接嵌进 SQL，有注入风险，只用于表名/列名等无法参数化的位置且必须白名单校验。

104.问：为什么要统一返回 Result？code 为什么不直接用 HTTP 状态码？
答：统一返回让前端用一个拦截器处理所有响应。HTTP 状态码表达传输层结果（404 没这地址、500 服务器炸），业务结果（用户名已存在、余额不足）在 HTTP 眼里都是 200——业务 code 和 HTTP 状态码是叠加关系不是替代关系。

105.问：全局异常处理器的原理？
答：@RestControllerAdvice + @ExceptionHandler。异常从 Service 穿透 Controller 落到 DispatcherServlet，它按异常类型匹配 Advice 里的处理方法，返回值照常序列化成 Result——保证报错也长统一返回的形状，业务代码里不用写 try-catch。

## 八、网络 / 认证

106.问：Cookie、Session、JWT 三种会话方案对比？
答：Cookie 存客户端、容量小、可被篡改；Session 存服务端安全但占内存、集群要做共享；JWT 把用户信息签名后放令牌里，服务端无状态、天然支持集群，代价是无法主动踢人、注销要额外手段。

107.问：JWT 由哪三部分组成？
答：Header（算法类型）.Payload（claims 数据如 userId、过期时间）.Signature（前两部分+密钥的签名）。前两段只是 Base64 编码不是加密——别放敏感数据；防篡改靠第三段签名。

108.问：JWT 怎么防篡改？
答：签名=对 header+payload 用服务端密钥做 HMAC。改任何一个字符，重算的签名对不上就抛异常拒绝。攻击者没有密钥就伪造不出合法签名——这也是密钥必须保密、必须从环境变量读的原因。

109.问：JWT 的缺点？怎么实现"退出登录"？
答：签发后到期前永远有效，服务端无法主动作废。解法：短有效期+刷新令牌；或维护黑名单（Redis 存已注销的 token，校验时先查）——本质是用一点"有状态"换控制力。

110.问：密码为什么用 BCrypt 而不是 MD5？
答：MD5 太快，彩虹表和暴力破解成本低，且同密码同哈希。BCrypt 自带随机盐（同密码每次哈希都不同）、故意慢（可调 cost），暴力破解成本指数级上升。校验用 checkpw 而不是比较哈希串。

111.问：TCP 为什么是三次握手，两次不行吗？
答：核心是双方都要确认"自己的发送和接收能力都正常"。两次时，服务端无法确认自己的发送能被客户端收到；且历史迟到的 SYN 会让服务端白建连接浪费资源。三次是可靠建立连接的最小次数。

112.问：四次挥手为什么是四次？
答：TCP 全双工，两个方向要分别关闭。被动方收到 FIN 时可能还有数据要发，所以 ACK 和自己的 FIN 分开发——比握手多一次。

113.问：TIME_WAIT 为什么要等 2MSL？
答：两个原因：保证最后一个 ACK 丢了对方重发 FIN 时还能应答（否则对方无法正常关闭）；让本连接的旧报文在网络里自然消亡，避免"上一段连接的迟到数据"串进新连接。

114.问：GET 和 POST 的区别？
答：语义上 GET 取数据（幂等、可缓存、参数在 URL）、POST 提交数据（不幂等、参数在 body）。浏览器地址栏只能发 GET——测 POST 接口必须用 Postman/Apifox。

115.问：401 和 403 的区别？
答：401 未认证——你是谁都不知道（没带 token/token 无效），该去登录；403 已认证但无权限——知道你是谁，但你不配（普通用户访问管理员接口）。

## 九、工程实践 / 分布式 / AI

116.问：为什么要分 Controller/Service/Mapper 三层？
答：各说各的语言：Controller 管 HTTP（参数接收、Result 封装），Service 管业务规则（判断、组合、事务），Mapper 管 SQL。改表不动 Controller、换接口形式不动业务——变化被隔离在单层内，也方便分层测试。

117.问：Entity/DTO/VO 为什么要分开？
答：Entity 映射表结构；DTO 接前端入参（只收该收的字段，防恶意传参）；VO 是出参（裁掉 password 等敏感字段）。实体直接进出接口=表结构和敏感数据裸奔，且入参出参变化频率完全不同。

118.问：什么是本地消息表（outbox）？解决什么问题？
答："改业务数据+发消息"无法跨库和 MQ 原子完成。outbox 把要发的消息作为一行记录和业务数据写进同一个本地事务，再由定时任务/订阅扫表投递到 MQ，投递成功标记完成——用"本地事务+重试"实现最终一致。

119.问："至少一次投递"下怎么保证业务不重复？
答：至少一次=必然可能重复，所以消费端必须幂等：用唯一业务键（如 idempotency_key 建唯一索引）插入判重，或 Redis setnx 记录已处理的消息 id——重复消息直接跳过。

120.问：幂等有哪些实现手段？
答：数据库唯一索引兜底（最可靠）；乐观锁版本号；Redis setnx 令牌；状态机（只允许特定状态流转）。对外接口让调用方带幂等键，服务端按键判重。

121.问：重试为什么必须"有限次数+退避+只对可重试错误"？
答：无限重试会把故障方打死（重试风暴）；固定间隔会同步冲击，指数退避+抖动错峰；参数错这类必然失败的错误重试毫无意义，只有超时/临时故障值得重试——且被重试的接口必须幂等。

122.问：降级和熔断的区别？
答：熔断是"自动开关"：下游错误率超阈值就断开，一段时间内直接快速失败，半开试探恢复——防止被拖死；降级是"有损服务"：核心保住、非核心返回兜底数据或关闭。熔断常触发降级，两者配合。

123.问：FastAPI 为什么快？
答：基于 ASGI（uvicorn）异步模型，IO 等待时事件循环切去处理别的请求，单进程高并发；配合 Pydantic 的高效校验序列化。对照：传统同步框架一个请求占一个线程死等 IO。

124.问：FastAPI 里同步 def 和 async def 路由的调度差异？
答：async def 跑在事件循环里，遇 await 让出，绝不能写阻塞调用（会卡死整个循环）；同步 def 会被扔进线程池执行，阻塞也不拖累循环。原则：函数里全是异步库用 async def，有阻塞库就用普通 def。

125.问：Pydantic 的校验发生在哪一层？
答：请求进入路由函数之前——框架按参数类型声明自动反序列化+校验，不合法直接 422 返回，函数体里拿到的一定是合法对象。对照 Spring 的 @RequestBody+@Valid。

126.问：FastAPI 的 Depends 和 Spring IOC 像在哪？
答：都是"我声明需要什么，框架负责造好给我"。Depends 按函数签名递归解析依赖并注入，还能做鉴权/分页这类可复用逻辑——相当于轻量版的依赖注入容器，作用域默认每请求一次。

127.问：RAG 的完整流程？
答：离线：文档清洗→切分（chunk）→embedding 向量化→入向量库。在线：用户问题向量化→混合检索（向量+BM25）→重排→拼进 prompt 交给 LLM 生成→带引用返回。评估用固定测试集看命中率和回答质量。

128.问：切分为什么要 overlap？
答：硬切会把一句话/一个语义单元拦腰斩断，答案恰好跨块时两块都检索不全。相邻块重叠一部分（如 10~20%）保证语义完整性，代价是存储和检索量略增。

129.问：混合检索为什么优于纯向量？
答：向量检索抓语义相似但对精确词（型号、编号、人名）不敏感；BM25 抓关键词精确匹配但不懂同义改写。两者互补，合并后再重排，召回率显著高于单路。

130.问：重排（rerank）解决什么问题？
答：召回阶段为了快用的是粗排（向量内积/BM25 分数），排序质量一般。重排用更强的交叉编码模型对 top-N 候选和问题逐对精算相关性，把真正相关的排到前面——少量候选上花大算力，性价比最高。

131.问：Agent 的四件套？
答：Planning（把目标拆成步骤）、Memory（短期对话上下文+长期知识）、Tools（调用外部能力：搜索/数据库/API）、Action（执行并根据结果调整）。没有工具调用的只能叫聊天机器人。

132.问：ReAct 循环是什么？
答：Reasoning+Acting 交替：想（分析当前该干嘛）→做（调工具）→看（观察结果）→再想……直到得出最终答案。比一次性生成靠谱，因为每步都基于真实反馈修正。

133.问：LangChain、LangGraph、Dify 一句话区别？为什么手写不用框架？
答：LangChain=组件库（模型/检索/工具的封装），LangGraph=状态图编排（复杂流程带循环分支），Dify=低代码平台（拖拽建应用）。学习期手写是为了吃透每一层原理、面试能讲清楚，且小项目框架反而是黑盒负担。

134.问：多轮对话的上下文怎么维护？
答：全量塞历史会爆 token：滑动窗口只留最近 N 轮；更早的做摘要压缩；关键事实抽取存长期记忆按需检索。原则：token 花在对当前回答有用的信息上。

135.问：Prompt 注入是什么？怎么防？
答：用户在输入里夹带"忽略以上指令"来劫持系统提示词。防：系统/用户消息严格分层、输入过滤和转义、工具调用白名单+最小权限、敏感操作二次确认、输出校验——默认把用户输入当不可信数据。

## 十、把项目讲成八股（面试话术弹药）

136.问：讲讲你项目的登录认证是怎么做的？
答：注册时 BCrypt 加盐哈希落库；登录校验通过后用 HS256 签发 JWT（payload 放 userId、exp），密钥从环境变量读；拦截器统一校验 token，解析出的 userId 放 ThreadLocal 供本次请求全程使用，afterCompletion 里 remove 防串号和泄漏；登录注册在放行白名单里。

137.问：为什么选 JWT 而不是 Session？
答：无状态：服务端不存会话，重启不掉线、水平扩容不需要共享存储；自带过期和防篡改。代价是无法主动作废，我用短有效期缓解。如果业务强依赖"踢人下线"，我会换 Redis 存会话或加黑名单。

138.问：拦截器里为什么用 ThreadLocal 传 userId，不用方法参数？
答：一次请求全程同一条线程，ThreadLocal 相当于线程私有储物格：拦截器存、任何层取，不污染所有方法签名，也不会像全局变量那样被并发请求互相覆盖。配套纪律是请求结束必须 remove——线程池会复用线程。

139.问：你的全局异常处理是怎么设计的？
答：自定义 UserException 表示"能给用户看的业务错误"，@RestControllerAdvice 里分两级接：UserException 返回它的 message，兜底 Exception 返回"服务器开小差"并记日志——保证任何异常出口都是统一 Result 格式，敏感堆栈不外泄。

140.问：接口的参数校验你是怎么做的？
答：分层校验：Controller 用注解校验格式（非空、长度、格式），Service 校验业务规则（用户名是否已存在）；错误统一走全局异常处理返回 Result。原则：不信任何前端传参，服务端必须重新校验。

141.问：数据库密码和 JWT 密钥怎么管理的？
答：不硬编码、不进 Git：本地用环境变量注入（yml 里 ${MYSQL_PASSWORD}），IDEA 运行配置和测试配置分别设置；上线后用服务器环境变量/密钥管理服务。公开仓库里泄露过的密钥一律视为作废轮换。

142.问：日志里打过密码或 token 吗？
答：明确禁止：日志只打 uri、userId、异常类名等定位信息；实体 toString 用 @ToString.Exclude 排除 password；排查探针打印变量前先过一遍"这是不是敏感数据"。

143.问：写测试你遵循什么套路？
答：每个接口至少一正一反：正常参数断言状态码+关键字段，异常输入断言 4xx/返回 null；MockMvc 公式 perform(请求).andExpect(断言)；测试数据固定化（id=1 永远是 user01）保证可重复；出 bug 先补一条能复现它的测试再修。

144.问：你怎么排查一个接口 401？
答：三种来源按序查：没带 token / token 是旧密钥签的或过期 / 路径没进放行名单。手段：Postman 看状态码，服务端看拦截器探针日志（拦截到什么路径、解析抛什么异常），catch 里绝不静默吞异常。

145.问：@Select 手写 SQL 和 BaseMapper 你怎么选？
答：单表 CRUD 用 BaseMapper 白送的（生成的 SQL 列名全展开，表加字段不受影响）；多表关联、复杂统计手写。我对比过两者生成的 SQL——手写灵活但要自己维护，MP 版规范但复杂场景表达不了。

146.问：为什么用 Flyway 管数据库脚本（而不是手工执行）？
答：表结构变更进版本控制，V1、V2 有序演进，任何人克隆项目启动即自动建库到最新版本——消灭"我这能跑你那缺表"的环境漂移，也杜绝 ddl-auto 自动改表的失控风险。

147.问：接口幂等你在项目里怎么落地？
答：工单创建带 idempotency_key 唯一索引，重复提交第二次插入违反唯一约束，捕获后返回第一次的结果——数据库层兜底最可靠，前端防重按钮只是体验优化不是保证。

148.问：压测数字你是怎么得到的？（简历数字的来源）
答：固定环境和数据量，用工具（如 JMeter）对目标接口阶梯加压，记录 QPS/P99；做一个优化（加索引/加缓存）就重测一轮，报告里写"优化前后对比+条件"——数字可复现才敢写简历。

149.问：自我介绍你的学习项目时的一句话框架？
答：做了什么（设备运维管理系统，含认证/RBAC/缓存/幂等）+ 技术栈（SpringBoot+MyBatis-Plus+MySQL+Redis）+ 亮点两个（可复现压测优化数字、测试覆盖关键路径）+ 我能讲清每一行代码为什么这么写。

150.问：面试官问到你不会的怎么办？
答：不装懂：先说结论"这个点我还没实践过"，再关联最近的已知（"但它和我用过的 X 类似，我理解是……"），最后表达路径（"下来我会用 X 方式验证"）。诚实+关联+行动，比背错答案加分。
